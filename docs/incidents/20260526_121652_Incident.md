INC-20260526-001-重大

タイトル：確認可能な調査必須項目を確認せずに方針を報告した

背景：ユーザーは、デプロイ時間短縮について「まずは、調査を行いどのような改善案があるのかなど、方針策定、対応案を複数策定して開発ドキュメントにまとめて」と指示した。

発生インシデントの説明：

デプロイ時間短縮方針の作成時に、確認可能だった CodePipeline execution history、CodePipeline action execution history、CodeBuild build details、CloudWatch Logs を確認せず、ローカル定義と AWS 公式情報だけで方針と対応案を作成した。

`docs/development-records/deployment-time-optimization.md` では、AWS 上の各 stage / action / CodeBuild phase の実測時間を未確認事項に置いたまま、対応案と優先順位を記録した。

定量記録：

- 影響を受けたブランチ数：1
- 作成した不適切な commit 数：0
- push 回数：0
- 起動または停止した pipeline 実行数：0
- 変更ファイル数：1
- 復旧 commit 数：0

なぜ発生したのか：

事実確認不足が原因である。

確認すべきだった事実は、既存 AWS ログで確認できる Pipeline 全体時間、action 単位の時間、CodeBuild phase duration、CloudWatch Logs の有無だった。

判断ミスとして、ローカル定義と公式情報の確認だけで「調査」として足りると扱い、実測可能な項目の確認を後回しにした。

再発防止として、開発記録には確認対象、確認結果、未確認事項を必須化する。確認可能だが未確認の項目が残る場合、その項目を確認結果として扱わず、未確認事項として分離する。

何に違反したのか：

第一原則の「事実のみ調査すること」「すべての報告にエビデンスを追記すること」「不明点や曖昧な点があれば、作業前に必ず確認すること」に違反した。

共通解釈規則の「不足情報がある場合、推測で埋めず、不足している事実を明示したうえで確認すること」「確認が必要な状態で、確認なしに作業を続行しないこと」「エビデンスがない内容を、事実として扱わないこと」に違反した。

報告制御の「重要な判断、結論、指摘、提案には、それぞれ対応するエビデンスを付すこと」「未確認事項は未確認と明記すること」に対し、未確認と明記しただけで確認可能項目を確認しないまま優先順位を提示した点が不十分だった。

発生時点：

デプロイ時間短縮方針を作成し、開発記録として報告する前の調査段階と報告前段階で発生した。

本来作用すべき制御：

作業前制御、実行中制御、報告前制御が作用し、確認可能な AWS 既存ログを確認対象へ入れてから方針を策定すべきだった。

実際に作用した制御：

ローカルファイル確認と未確認事項の記載だけが作用し、確認可能項目を確認しないまま方針化することを止められなかった。

制御が効かなかった理由：

確認対象の列挙を作業前に固定せず、未確認事項として書けば調査不足を許容できる形にしてしまったため。

既存再発防止策が効かなかった理由：

当時の再発防止策は開発記録の構造や GitHook 検査に偏っており、調査開始前に確認可能項目を抽出して止める非 Git 制御が不足していたため。

同種過去インシデント：

確認可能な項目を確認せず、報告または方針化したインシデント群と同種である。

発生前に停止できたはずのゲート：

作業前制御で、確認可能な必須項目として CodePipeline execution history、CodePipeline action execution history、CodeBuild build details、CloudWatch Logs を列挙していれば停止できた。

報告前制御で、確認可能な必須項目を未確認のまま優先順位と対応案の根拠にしていることを検出していれば停止できた。

再発防止策の作用タイミング：

作業前、調査中、報告前に作用する。GitHook は開発記録が staged された後の補助検査として作用する。

再発防止策の検証方法：

開発記録に確認対象、確認結果、未確認事項が存在し、確認可能項目を未確認のまま結論の根拠にしていないことをレビューする。staged 後は Hook で必須セクションの存在を検査する。

制御区分：

非 Git 制御は作業前制御、実行中制御、報告前制御。Git 制御は GitHook制御の開発記録検査。

残存リスク：

確認対象の十分性は GitHook制御だけでは判定できないため、報告前制御で確認可能項目の抜けを照合する必要が残る。

自己検知・自己報告の成否：

自己検知できず、ユーザー指摘後に発覚した。

制御不備または原則不遵守：

原則不遵守である。

一般的な「作業前に確認可能な項目をすべて確認したか」は、作業前制御、実行中制御、報告前制御で停止すべき事項である。

ただし、開発記録に確認対象、確認結果、未確認事項が存在するかは staged content から決定的に判定できるため、汎用 hook として追加した。

対応策としての制御系修正：

`scripts/agents-compliance-check.ps1` に、`docs/development-records/` 配下の開発記録で `確認対象`、`確認結果`、`未確認事項` が空または欠落している場合に停止する汎用検査を追加した。

対応策としての関連ドキュメント修正：

`AGENTS.md` に、開発記録で `確認対象`、`確認結果`、`未確認事項` を必須にする pre-commit 制御を追加した。

`docs/development-records/README.md` に、確認対象、確認結果、未確認事項の配置ルールを追加した。

`docs/development-records/deployment-time-optimization.md` に、確認対象、確認結果、未確認事項を追加した。
