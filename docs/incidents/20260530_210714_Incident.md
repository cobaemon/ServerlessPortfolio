INC-20260530-001-違反

タイトル：ドキュメント反映状況の未調査報告

背景：ユーザーは「各種ドキュメントには反映済みか」と質問した。この質問は、新しいデプロイ仕様がどのプロジェクトドキュメントへ反映済みかを、対象ドキュメントを確認したうえで報告する必要がある内容だった。

発生インシデントの説明：

対象ドキュメントを `rg` やファイル閲覧で確認せず、既知の作業記憶だけに基づいて「全ドキュメントへの反映済みとは言えません」と報告した。報告内で `docs/deployment.md`、`docs/ai-progress/README.md`、`docs/index.md` などを未確認と記載したが、未確認のまま回答を完了した。

定量値：影響を受けたブランチ数は 0。作成した不適切な commit 数は 0。push 回数は 0。起動または停止した pipeline 実行数は 0。変更ファイル数は、本記録と再発防止策実装前の時点で 0。復旧 commit 数は本記録作成時点で 0。

なぜ発生したのか：

質問対象である「各種ドキュメント」を、確認対象として列挙しなかった。報告前制御で、報告対象がユーザーの質問と一致しているか、重要な判断に確認済み事実が対応しているかを確認しなかった。確認可能なファイル確認を実施しないまま、未確認事項を含む回答を返した。

事実確認不足：`docs/deployment.md`、`docs/current-state.md`、`docs/index.md`、`docs/staging-deployment-runbook.md`、`README.md`、`docs/development-records/deployment-time-optimization.md` を回答前に確認しなかった。

確認すべきだった事実：新しいデプロイ仕様が反映済みのドキュメント、未反映のドキュメント、対象外のドキュメント、対象外と判断する根拠。

何に違反したのか：

第一原則の「事実のみ調査すること」「事実のみ報告すること」「すべての報告にエビデンスを追記すること」に違反した。共通解釈規則の「エビデンスがない内容を、事実として扱わないこと」「確認が必要な状態で、確認なしに作業を続行しないこと」に違反した。報告制御の「重要な判断、結論、指摘、提案には、それぞれ対応するエビデンスを付すこと」「未確認事項は未確認と明記すること」に違反した。報告前制御の「重要な判断、結論、指摘、提案に確認済み事実が対応していること」に違反した。

発生時点：

ユーザーの「各種ドキュメントには反映済みか」という質問に回答する直前。

本来作用すべき制御：

応答前制御で質問対象を「新しいデプロイ仕様の各種ドキュメント反映状況」と固定する必要があった。作業前制御で、回答に必要な確認対象ドキュメントを列挙する必要があった。ツール実行前制御で、ファイル確認が質問回答に必要であることを確認する必要があった。報告前制御で、未確認のまま完了扱いしていないかを停止する必要があった。

実際に作用した制御：

非 Git の応答前制御と報告前制御は自己点検として存在したが、実際には停止しなかった。GitHook制御はファイル変更がなかったため作用しなかった。

制御が効かなかった理由：

報告前に「各種ドキュメント」という確認対象を機械的または手順上で列挙しなかった。GitHook は staged diff を対象にするため、チャット回答のみの未調査報告を停止できなかった。

既存再発防止策が効かなかった理由：

既存の開発記録ルールには、開発記録を恒久的な仕様または運用手順として採用する場合に `docs/` 直下へ反映する記述があった。しかし、正式採用記録にドキュメント反映状況を必須記録させる機械的条件がなく、報告前に各種ドキュメント反映状況を検査する具体条件も不足していた。

同種過去インシデント：

`docs/incidents/20260526_121652_Incident.md` は、デプロイ時間短縮方針作成時に確認可能だった AWS 既存ログを確認しなかった事象を記録している。確認可能項目を確認せず報告した点で同種である。

発生前に停止できたはずのゲート：

応答前制御で質問対象を固定するゲート、作業前制御で確認対象を列挙するゲート、ツール実行前制御で必要なファイル確認を実施するゲート、報告前制御で確認済み事実と報告内容の対応を確認するゲート。

再発防止策の作用タイミング：

応答前制御と報告前制御は、ドキュメント反映状況や仕様反映状況を質問された時点で作用する。GitHook制御は、仕様、運用、手順、デプロイ、pipeline、外部環境に関する正式採用または本番反映を開発記録へ staged する時点で作用する。

再発防止策の検証方法：

`scripts/agents-compliance-check.ps1` の PowerShell 構文解析が成功することを確認する。正式採用または本番反映を含む開発記録に `ドキュメント反映状況` がない場合、pre-commit hook の `Assert-AdoptedSpecRecordsIncludeDocumentReflectionStatus` が停止することを確認する。新しいデプロイ仕様の反映状況報告では、対象ドキュメントを確認し、反映済み、未反映、対象外を根拠付きで報告する。

制御区分：

応答前制御、作業前制御、ツール実行前制御、報告前制御、GitHook制御、事後監査制御の複合制御。GitHook だけではチャット回答を停止できないため、非 Git 制御で質問回答前の調査を必須化し、GitHook では正式採用記録のドキュメント反映状況漏れを停止する。

残存リスク：

チャット回答だけで完結する質問では GitHook が作用しない。残存リスクに対しては、応答前制御と報告前制御で、確認可能な対象を列挙してから回答する必要がある。

自己検知・自己報告の成否：

失敗。ユーザーから「なぜ確認せず未確認で報告している？」と指摘されるまで、必要な調査を実施していないことを自己検知して報告できなかった。

対応策としての制御系修正：

`AGENTS.md` の pre-commit 制御に、`docs/development-records/` 配下の開発記録で、仕様、運用、手順、デプロイ、pipeline、外部環境に関する正式採用、本番反映、恒久化を記録する場合、`ドキュメント反映状況` が存在しなければ停止する条件を追加した。`scripts/agents-compliance-check.ps1` に同条件を検査する `Assert-AdoptedSpecRecordsIncludeDocumentReflectionStatus` を追加した。

対応策としての関連ドキュメント修正：

`docs/development-records/README.md` に、仕様、運用、手順、デプロイ、pipeline、外部環境に関する開発記録を正式採用または本番反映した場合、同じ記録内に `ドキュメント反映状況` を設け、反映済み、未反映、対象外の区分と対象ドキュメントを記載するルールを追加した。`docs/deployment.md`、`docs/current-state.md`、`docs/development-records/deployment-time-optimization.md` に新しいデプロイ仕様と正式採用後の反映状況を記録した。

復旧案：

案A：新しいデプロイ仕様の反映状況を実際に確認し、未反映の恒久ドキュメントへ追記し、開発記録に正式採用後のドキュメント反映状況を記録する。案Aを実施する。

案B：ドキュメント反映を行わず、チャットで未反映一覧だけを報告する。恒久的な運用仕様が残らないため採用しない。
