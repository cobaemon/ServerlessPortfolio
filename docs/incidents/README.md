# インシデント記録

このディレクトリは、原則への直接違反または間接違反が発生した事象を記録します。

インシデントは、感情的または曖昧な分類ではなく、原則に違反しているかどうかで判定します。

インシデントレベルは、原則違反が発生した後の影響度によって分類します。

## 配置ルール

原則への直接違反または間接違反、指示違反、手順逸脱、影響発生、復旧対応の記録を配置します。

プロジェクト仕様、設計、設定、運用、デプロイ、開発環境など、継続的に参照するプロジェクトドキュメントは `docs/` 直下に配置します。

AI 作業進捗、作業途中の AI 用 TODO、AI が使用する確認記録は `docs/ai-progress/` に配置します。

開発過程で発生または使用する調査、方針、対応案、検討記録は `docs/development-records/` に配置します。

インシデント記録を作成する場合は、この README のファイル名、インシデント番号、インシデントレベル、記録形式に従います。

## インシデント対応サイクル

インシデント対応は、記録ファイルの作成だけでは完了しません。

該当インシデントの対応策として、次を責任範囲に含めます。

- インシデント記録ファイルを作成する。
- 発生した事実を、確認済み事実、未確認事項、影響範囲に分離する。
- 違反した指示、手順、停止条件、完了条件を特定する。
- 発生時点と、その時点で本来作用すべき制御を特定する。
- 実際に作用した制御と、作用しなかった制御を分離する。
- 既存再発防止策がなぜ効かなかったかを特定する。
- 同種過去インシデントの有無を確認する。
- 発生前に停止できたはずの応答前制御、作業前制御、ツール実行前制御、ファイル編集前制御、実行中制御、報告前制御、事後監査制御のいずれかを特定する。
- 再発防止策の作用タイミングと検証方法を特定する。
- GitHook で決定的に停止できる違反は、`.githooks` または `scripts/agents-compliance-check.ps1` の修正要否を確認する。
- GitHook で停止できない違反は、応答前制御、作業前制御、ツール実行前制御、ファイル編集前制御、実行中制御、報告前制御、事後監査制御のどこで止めるかを記録する。
- commit、branch-finalize、push、deploy の対象範囲に関するインシデントでは、元の保留変更、実際に反映した変更、反映しなかった変更、ユーザー確認の有無、復旧案を記録する。
- 制御系ファイルの修正は、ユーザーが制御系ファイル変更を明示指示した場合に限る。インシデント記録の作成だけを理由に、制御系ファイル修正を自動要求してはならない。
- 手順書、運用ドキュメント、関連ドキュメントに不足または曖昧さがある場合は修正する。
- 制御系修正または関連ドキュメント修正が不要、または実装不能な場合は、その理由をインシデント記録に明記する。

インシデント対応を完了扱いにできるのは、インシデント記録、発生時点、本来作用すべき制御、実際に作用した制御、制御が効かなかった理由、既存再発防止策が効かなかった理由、同種過去インシデント、発生前に停止できたはずのゲート、再発防止策の作用タイミング、再発防止策の検証方法、制御区分、残存リスク、自己検知・自己報告の成否、制御系修正要否、関連ドキュメント修正要否の確認がすべて記録されている場合に限ります。

## ファイル名

インシデント記録ファイルは次の形式で作成します。

```text
{作成時のタイムスタンプ}_Incident.md
```

タイムスタンプは Windows ファイル名で使用できる `yyyyMMdd_HHmmss` 形式とします。

## インシデント番号

インシデント番号は次の形式で採番します。

```text
INC-{yyyyMMdd}-{3桁連番}
```

同一日に複数発生した場合は、作成順に `001` から 1 ずつ増やします。

## インシデントレベル

インシデントレベルは次の降順で分類します。

| レベル | 分類基準 |
| --- | --- |
| 致命的 | 復元不可 |
| 侵害 | 実際に実害が発生 |
| 重大 | 複数の違反が同時多発 |
| 違反 | 単発の違反 |

staging、prod、IAM、CloudFormation、CodePipeline、S3、CloudFront、Route53、ACM などの実環境リソースが、未コミット変更、未承認手順、または誤った手順によって実際に変更された場合は実害として扱い、少なくとも `侵害` に分類します。

## 記録形式

各インシデントファイルは次の形式で記録します。

```text
インシデント番号-インシデントレベル

タイトル：{インシデント概要}

背景：{指示や要件}

発生インシデントの説明：

なぜ発生したのか：

何に違反したのか：

発生時点：

本来作用すべき制御：

実際に作用した制御：

制御が効かなかった理由：

既存再発防止策が効かなかった理由：

同種過去インシデント：

発生前に停止できたはずのゲート：

再発防止策の作用タイミング：

再発防止策の検証方法：

制御区分：

残存リスク：

自己検知・自己報告の成否：

対応策としての制御系修正：

対応策としての関連ドキュメント修正：

復旧案：
```

## 定量記録項目

発生インシデントの説明には、確認できる範囲で次の数値を含めます。

- 影響を受けたブランチ数
- 作成した不適切な commit 数
- push 回数
- 起動または停止した pipeline 実行数
- 変更ファイル数
- 復旧 commit 数

## 定性記録項目

発生インシデントの説明、なぜ発生したのか、何に違反したのかには、確認できる範囲で次を含めます。

- 逸脱した指示または手順
- 発生した影響
- 復旧方法
- 残存リスク
- 発生時点
- 元の保留変更
- 実際に反映した変更
- 反映しなかった変更
- ユーザー確認の有無
- 復旧案
- 本来作用すべき制御
- 実際に作用した制御
- 制御が効かなかった理由
- 既存再発防止策が効かなかった理由
- 同種過去インシデント
- 発生前に停止できたはずの応答前制御、作業前制御、ツール実行前制御、ファイル編集前制御、実行中制御、報告前制御、事後監査制御
- 再発防止策の作用タイミング
- 再発防止策の検証方法
- 制御区分
- 自己検知・自己報告の成否
- 再発防止のために必要な制御
- 対応策として実施した制御系修正、または修正不要と判断した理由
- 対応策として実施した関連ドキュメント修正、または修正不要と判断した理由
