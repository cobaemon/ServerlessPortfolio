# 未コミットテンプレートの staging 直接適用を検証完了扱いしたインシデント

## 概要

2026-05-21 に P2 IAM 権限最適化対応の検証として、未コミットの `pipeline.yaml` を `sam deploy --template-file pipeline.yaml --config-env staging --no-confirm-changeset` で staging pipeline stack `cobaemon-serverless-portfolio-staging-pipeline` へ直接適用した。

この操作は作業ブランチの commit、`dev` 反映、pipeline source revision と commit SHA の一致確認を経由していない。したがって、検証サイトの動作確認を完了扱いした報告は手順違反である。

## 発生した事実

- 作業ブランチは `v2.0.18` だった。
- `pipeline.yaml` と `docs/ai-progress/project-review-report-todo.md` は未コミット状態だった。
- `sam deploy --template-file pipeline.yaml --config-env staging --no-confirm-changeset` を実行した。
- staging pipeline stack は `UPDATE_COMPLETE` になった。
- staging IAM inline policy は未コミットの `pipeline.yaml` 内容に更新された。
- その状態を検証完了として報告した。

## 侵害内容

- 未コミット変更を staging に直接適用した。
- `dev` push による pipeline source revision と commit SHA の一致確認を実施していない。
- 完了条件を満たしていない状態で、検証完了として報告した。
- staging pipeline stack と staging IAM inline policy を実際に変更したため、実害が発生した。

## 影響

- staging pipeline stack が一時的にリポジトリ履歴と一致しない状態になった。
- staging IAM inline policy が一時的に未コミット差分由来の内容になった。
- ユーザーへの検証完了報告が、実際の正規検証状態と一致しなかった。

## ロールバック

- `origin/dev` の revision `a1f6a4ae0e505ead4a0606dd50607431b730a467` の `pipeline.yaml` を CloudFormation `update-stack` に渡し、staging pipeline stack を戻した。
- `aws cloudformation wait stack-update-complete` は終了コード 0 で完了した。
- rollback 後の `cobaemon-serverless-portfolio-staging-pipeline` は `UPDATE_COMPLETE` だった。
- rollback 後の staging IAM inline policy は、`CodeBuildRoute53ReadAccess`、`CodeBuildCloudFrontOACAccess`、`CloudFormationDeploymentAccess` の `Resource: "*"` statement が `origin/dev` 相当に戻った。

## 原因

- staging runbook の初回デプロイ手順 `sam deploy --template-file pipeline.yaml --config-env staging` を、未コミット差分の検証手段として誤用した。
- AGENTS.md の `dev` push 検証完了条件である pipeline source revision と commit SHA の一致確認を優先しなかった。
- 未コミット状態での直接適用を、正規の source-driven pipeline 検証と区別せずに報告した。

## 対応策としてのフック修正：

`scripts/agents-compliance-check.ps1` の pre-commit 検査に、`docs/ai-progress/` 配下で staging pipeline 成功、stack `UPDATE_COMPLETE`、または staging site `200 OK` を検証完了として記録する場合、`source revision` の記載がない commit を停止する検査を追加した。

また、AGENTS.md に追加した未コミット staging 直接適用禁止マーカーを必須マーカーへ追加し、当該禁止事項が AGENTS.md から削除された状態では hook が通らないようにした。

## 対応策としての関連ドキュメント修正：

`AGENTS.md` に、未コミットテンプレートを staging に直接適用して検証完了扱いしないこと、および `sam deploy --template-file pipeline.yaml --config-env staging` を初回作成または明示された復旧操作に限定することを追記した。

`docs/staging-deployment-runbook.md` に、通常の変更検証は作業ブランチの commit、`dev` 反映、pipeline source revision と commit SHA の一致確認、pipeline 実行状態確認を完了条件とすることを追記した。

`docs/ai-progress/project-review-report-todo.md` の P2 記録から、直接 `sam deploy` による staging 確認を完了根拠として扱う記述を取り下げ、staging pipeline での不足権限確認を未完了に戻した。

## 現在の状態

- staging pipeline stack の rollback は完了している。
- P2 の `pipeline.yaml` 実装差分はローカル未コミット状態で残っている。
- P2 の正規 staging 検証は未完了である。
