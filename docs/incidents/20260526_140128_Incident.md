INC-20260526-004-重大

タイトル：質問を作業指示として扱い、確認責任範囲と再発防止策を逸脱した

背景：ユーザーは、`staticfiles` のローカル確認について「なぜ検証しない？」と質問した。ユーザーはその時点で追加作業の明示指示を出していなかった。

発生インシデントの説明：

ユーザーの質問に対し、質問への回答ではなく、ローカルブラウザ検証を開始しようとした。

その前段で、`staticfiles` を Git 管理外にした場合の確認作業として、再生成、manifest 検査、差分確認までは行ったが、再生成後の `index.html` をローカルで配信し、ブラウザで CSS、JavaScript、画像、favicon、font、console error を確認するところまで実施しなかった。

その後の回答では、公開サイトという表現から staging / S3 / CloudFront 検証へ短絡し、Docker ローカル環境での検証という開発手法上の確認段階を説明から外した。

さらに、以前に「再発防止策は特定インシデントに固有化せず、普遍的かつ汎用的に設計する」ことを記録・実装した後にも、質問と作業指示の区別、確認責任範囲、スコープ変更を実行前に止められなかった。

定量記録：

- 影響を受けたブランチ数：1
- 作成した不適切な commit 数：0
- push 回数：0
- 起動または停止した pipeline 実行数：0
- 変更ファイル数：0
- 復旧 commit 数：0

なぜ発生したのか：

事実確認不足と実行前制御違反が原因である。

確認すべきだった事実は、ユーザーの発話が質問であり、追加検証実行の明示指示ではないこと、ローカル Docker 環境が staging に出す前の検証手段として存在すること、`staticfiles` 管理外化の確認範囲には生成だけでなくローカル配信とブラウザ確認が含まれることだった。

元の指示範囲は、`staticfiles` の目的、管理外化の変化、確認作業の実施、そしてその確認結果への質問回答だった。

不適切に狭めた範囲は、生成と manifest 検査までを確認作業として扱い、ローカル配信とブラウザ確認を未実施のまま結論を出したことだった。

不適切に拡張した範囲は、質問に対する回答を追加作業実行へ読み替えたことだった。

再発防止として、AGENTS と hook のレビューを行い、インシデント関連の制御が、質問と作業指示の区別、確認対象・確認結果・未確認事項、スコープ変更、外部環境検証、インシデント記録要件をどこまで扱えるかを洗い出す。

何に違反したのか：

第一原則の「不明点や曖昧な点があれば、作業前に必ず確認すること」「明示的に許可がない限り、作業を行わないこと」に違反した。

共通解釈規則の「不足情報がある場合、推測で埋めず、不足している事実を明示したうえで確認すること」「確認が必要な状態で、確認なしに作業を続行しないこと」に違反した。

スコープ変更禁止の「ユーザーの明示指示を、一部の語句、単一の違反類型、または作業者に都合のよい範囲へ狭めてはならない」に違反した。

発生時点：

ユーザーの質問に回答する前の応答前段階、ローカルブラウザ検証を開始しようとした作業前段階、確認結果を報告した報告前段階で発生した。

本来作用すべき制御：

応答前制御で質問と作業指示を分離し、報告前制御でローカル検証責任範囲を生成と manifest 検査に狭めていないか確認すべきだった。

実際に作用した制御：

生成確認と manifest 検査だけが作用し、ブラウザ表示確認までを責任範囲として保持する制御は作用しなかった。

制御が効かなかった理由：

確認作業の完了条件を、生成物の作成成功に縮小して扱ったため。

既存再発防止策が効かなかった理由：

既存策は記録や Hook 検査に偏り、質問応答前と報告前に責任範囲を照合する制御として作用しなかったため。

同種過去インシデント：

質問を作業指示として扱ったインシデント、確認可能項目を確認せず報告したインシデントと同種である。

発生前に停止できたはずのゲート：

応答前制御で、ユーザー発言が質問であり追加作業指示ではないことを固定していれば停止できた。

作業前制御で、ローカルブラウザ検証を開始する明示許可がないことを確認していれば停止できた。

報告前制御で、ローカル検証責任範囲を生成と manifest 検査に狭めていることを検出していれば停止できた。

再発防止策の作用タイミング：

応答前、作業前、報告前に作用する。GitHook は記録や文書化後の補助検査に限定される。

再発防止策の検証方法：

ユーザー発言が質問か作業指示かを明示し、確認作業の完了条件がユーザーの責任範囲と一致していることを報告前に確認する。

制御区分：

非 Git 制御は応答前制御、作業前制御、報告前制御。GitHook制御はインシデント記録の事後監査項目検査。

残存リスク：

会話中の責任範囲解釈は Hook では発生前停止できないため、応答前と報告前の自己監査が必要である。

自己検知・自己報告の成否：

自己検知できず、ユーザー指摘後に発覚した。

制御不備または原則不遵守：

制御不備および原則不遵守である。

質問を作業指示として扱う違反は、会話解釈と実行前制御の違反であり、応答前制御と作業前制御で停止すべきである。

ただし、確認記録、開発記録、外部環境検証記録、インシデント記録については staged content から検査できるため、既存の汎用 GitHook制御が該当範囲をどこまでカバーしているかをレビューする。

対応策としての制御系修正：

今回の質問誤読そのものは GitHook制御ではなく、応答前制御と作業前制御で停止する。

GitHook制御には、開発記録の `確認対象`、`確認結果`、`未確認事項` の必須化、外部環境の検証完了主張の確認文脈必須化、インシデント記録の監査項目必須化が存在する。非 Git 制御は応答前制御、作業前制御、報告前制御として扱う。

対応策としての関連ドキュメント修正：

`AGENTS.md` に制御系0設計、応答前制御、作業前制御、ツール実行前制御、ファイル編集前制御、実行中制御、報告前制御、事後監査制御を追加し、原則を GitHook制御の有無に関係なく常時適用する制御として明文化する。
