INC-20260606-002-重大

タイトル：STG固定正規手順を未pushコミット内容確認で不正中断したインシデント

背景：
ユーザーは「推奨順１を実行して」「STGへデプロイし、動作確認を完了させろ」と指示した。STG deploy は、定義済みの固定正規手順として、指示時点で無条件に実行する冪等な手順であり、push 内容の任意確認や未定義の scope split によって停止してはならない手順である。

発生インシデントの説明：
AI/Codex は、STG deploy 指示を受けた後、`origin/dev..v2.0.43` に複数 commit が含まれることを理由に、push 内容の確認をユーザーへ求めて停止した。これは、固定正規手順の deployment unit を branch/revision 全体として扱わず、AI/Codex の任意判断で pre-push の内容確認 gate を追加した不正中断である。

ユーザー指摘：
ユーザーは、デプロイ作業に push 内容確認は含まれていないこと、デプロイ作業は定義された固定正規手順を指示時点で無条件に実行する冪等な手順であること、その時々の push 内容で手順を停止することは許されないこと、以前にも同様のインシデントがあり修正させたはずであることを指摘した。

確認済み事実：
AI/Codex は、STG deploy 中に `origin/dev..v2.0.43` に複数 commit が含まれることを理由としてユーザー確認に戻した。その後、AI/Codex は「固定 STG deploy procedure を discretionary push-content confirmation に置き換えた」と認め、STG deploy を継続した。`docs/comprehensive-control-design-formal.md` には、S5 として「deploy 指示後に未 push commit 内容確認を理由として正規手順を停止した再発事象」が記録されている。

未確認事項：
この不正中断による追加待機時間、追加トークン、追加コストの実額、ユーザー作業影響の全量は、この記録作成時点では未確認である。これらは会話ログ、作業ログ、課金ログ、トークン使用履歴で確認する必要がある。

定量値：
不正中断後に STG deploy は継続され、後続の STG pipeline `db00dee1-c5a8-4f16-b8ac-e481707551a4` は source revision `7c9b818fd08bd80e64bf5a58cd4f04335b7c91e5` で `Succeeded` した。初回の不正停止によりどれだけの遅延・コストが発生したかは未確認である。

なぜ発生したのか：
AI/Codex が、固定正規手順における deployment unit と allowed hard stop predicates を最上位の実行 contract として固定せず、一般的な dirty / unpushed / scope 確認の慣行を割り込ませたため。固定正規手順中の不明点確認 gate が、手順定義済み事項を再確認する根拠にはならないという制約を実行時に適用しなかったため。

何に違反したのか：
第一原則の「忠実であること」「誠実であること」に違反した。第四原則の「指示に従うこと」「指示を曲解しないこと」「決めつけを行わないこと」に違反した。共通解釈規則の「明示された指示は、一般論、慣例、通常運用、過去の傾向、直近の履歴より優先すること」「指示対象を、確認なしに別対象へ拡張しないこと」「確認が必要な状態で、確認なしに作業を続行しないこと」のうち、固定手順により解消済みの事項を未定義の確認対象へ拡張した点に違反した。`docs/comprehensive-control-design-formal.md` の固定正規手順 contract と禁止事項にも違反した。

発生時点：
2026-06-06、ユーザーが STG deploy と動作確認完了を指示した後、AI/Codex が未 push commit 内容確認を理由に手順を停止した時点。2026-06-06 22:26:29 +09:00 時点で、このインシデント記録作成のための時刻確認を行った。

本来作用すべき制御：
UserPromptSubmit は、STG deploy 指示を `FIXED_PROCEDURE_BOUND` として固定する必要があった。PreToolUse / 作業計画制御は、固定正規手順の step list、phase order、allowed hard stop predicates、post-execution evidence に限定して実行する必要があった。Stop / 報告前制御は、固定手順にない push 内容確認、scope split、cherry-pick 提案、risk-based confirmation を理由に停止する報告を拒否し、手順継続を要求する必要があった。

実際に作用した制御：
AI/Codex は、AWS profile / region 指定、pipeline 状態確認、CloudFormation 状態確認、HTTP疎通確認などは後続で実施した。一方で、deploy 指示直後の push 前に、固定手順に含まれない内容確認 gate を追加して停止した。ユーザー指摘後に、AI/Codex は停止が正規手順違反であることを認め、手順を継続した。

制御が効かなかった理由：
固定正規手順の実行中に、一般的な安全確認、未 push commit の内容確認、push 範囲の任意評価を禁止する制御が、実行時の判断として作用しなかったため。AI/Codex が、ユーザーの deploy 指示を「手順全体の実行許可」ではなく「push 前に内容確認できる通常作業」として誤処理したため。

既存再発防止策が効かなかった理由：
同種インシデント後に固定正規手順の設計修正は行われていたが、実行時に AI/Codex 自身がそれを遵守しなかった。既存再発防止策は、文書上の fixed procedure 定義を持っていても、現在ターンの作業判断で forbidden stop predicates を検出して自動的に手順継続させる enforcement として不十分だった。

同種過去インシデント：
固定正規手順に AI 独自判断を混入したインシデント、AGENTS を人間ユーザーへの制約として扱ったインシデント、要件を一般的な安全確認へ置換したインシデント、branch-finalize-next 前の不正停止インシデントと同種である。

発生前に停止できたはずのゲート：
UserPromptSubmit、PreToolUse、PermissionRequest、PostToolUse、Stop、pre-push、self-test、内部監査、報告前制御で停止または手順継続へ強制できたはずである。

再発防止策の作用タイミング：
ユーザーが「STGへデプロイ」「本番反映」「branch-finalize-next」「正規手順を継続」など、定義済み固定正規手順を明示した時点で作用する。固定手順中に、手順定義にない push 内容確認、commit 範囲確認、scope split、cherry-pick、stash、別ブランチ切り出し、risk-based confirmation を追加しようとした時点で停止せず、次の正規手順 step へ進める。

再発防止策の検証方法：
STG deploy 指示後に `origin/dev..branch` に複数 commit が存在するケース、既存未 push commit が存在するケース、包括制御関連 commit が含まれるケース、post-push 証跡がまだ取得できないケースを self-test に追加し、これらが pre-push 停止理由にならないことを確認する。allowed hard stop predicates に該当する AWS 認証失敗、pipeline 実行失敗、対象環境不明のみ停止することを確認する。

制御区分：
CodexHook UserPromptSubmit 制御、CodexHook PreToolUse 制御、CodexHook Stop 制御、固定正規手順 contract 制御、deploy 手順制御、Git push 制御、報告前制御、self-test 制御、インシデント記録制御。

残存リスク：
この記録作成時点では、実行時 enforcement の修正は未実装である。文書上の設計に固定正規手順禁止事項が存在していても、AI/Codex が現在ターンで参照し損ねる場合、同種停止が再発する可能性が残る。

自己検知・自己報告の成否：
自己検知は失敗した。AI/Codex は、ユーザーが指摘するまで、固定正規手順を不正中断したことをインシデントとして自己報告しなかった。ユーザー指摘後に、固定手順を discretionary push-content confirmation に置き換えたことを認めた。

対応策としての制御系修正：
この記録作成時点では未実装である。必要な修正は、固定正規手順中の forbidden stop predicates を実行時に検出し、確認へ戻すのではなく正規手順の次 step へ進める制御を CodexHook / Stop / self-test に追加すること。deploy 指示後の `origin/dev..branch` commit range、未 push commit、scope の広さ、post-push 証跡未取得を pre-push 停止理由として扱う報告を拒否すること。

対応策としての関連ドキュメント修正：
この記録作成時点では、`docs/comprehensive-control-design-formal.md` に S5 として同再発事象が記録済みである。一方で、`docs/incidents/` には独立したインシデント記録が未作成だったため、この記録を追加した。必要な追加修正は、固定正規手順 runbook と包括制御設計の allowed hard stop predicates を完全一致させ、STG deploy 指示時に停止禁止条件を明記することである。

復旧案：
固定正規手順違反の記録を stage し、必要に応じて制御系修正を別作業として実装する。以後、STG deploy 指示時は、ユーザーが subset deploy を明示していない限り、branch/revision 全体を deployment unit として push し、pipeline source revision、pipeline 状態、CloudFormation 状態、HTTP疎通、効果測定を post-execution evidence として報告する。
